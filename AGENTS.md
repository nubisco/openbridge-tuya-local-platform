This file helps AI coding agents understand the structure, conventions, and workflows of this project.

## Project Overview

**openbridge-tuya-local-platform** is an OpenBridge plugin that controls Tuya-based smart devices over LAN without cloud dependency. It communicates directly with devices using their Device ID and Local Key via the Tuya protocol (versions 3.1, 3.3, 3.4).

- **Plugin name**: `@nubisco/openbridge-tuya-local-platform`
- **Platform alias**: `TuyaLocalPlatform`
- **Package name**: `@nubisco/openbridge-tuya-local-platform`
- **License**: MIT

The plugin supports two loading modes in OpenBridge:

1. **Native mode** (`plugins[]`): loaded as a first-class OpenBridge plugin using `definePlugin()`. Creates its own standalone HAP bridge with hap-nodejs. Users pair this bridge separately in the Apple Home app.
2. **Openbridge compat mode** (`platforms[]`): loaded through OpenBridge's Openbridge compatibility shim. Uses OpenBridge's shared HAP bridge. Recommended for users who want all devices under a single bridge.

## Tech Stack

- **Language**: TypeScript (strict mode, `strictNullChecks` and `strictFunctionTypes` off)
- **Module system**: CommonJS (output to `dist/`)
- **Target**: ES2022
- **Runtime**: Node.js >= 20, OpenBridge >= 0.1.0
- **Config validation**: Zod (validates device config at plugin startup)

## Architecture

### Dual Export

`src/index.ts` uses a dual-export strategy so the single compiled file works on both OpenBridge loading paths:

```
require('./dist/index.js')         → module.exports.default = openbridgeSetup (function)  ← compat loader
import('./dist/index.js')          → mod.default = { manifest, setup, start, stop, ... }  ← native loader
```

`module.exports` is an object spread from the native plugin definition plus a `default` property pointing to the Openbridge setup function. Both loaders use `mod.default ?? mod`, so each gets what it needs.

### Native Plugin Lifecycle

When loaded via `plugins[]`:

1. `setup(ctx)` — validates config with Zod. Throws on invalid config so the daemon surfaces the error immediately.
2. `start(ctx)` — loads hap-nodejs from the host, creates a standalone HAP bridge, builds a minimal OpenbridgeAPI shim (`createOpenbridgeShim`), instantiates `TuyaLocalPlatform` with the shim, and publishes the bridge.
3. `stop(ctx)` — unpublishes the HAP bridge cleanly.

### Openbridge Compat Lifecycle

When loaded via `platforms[]`:

The standard Openbridge registration flow: `openbridgeSetup(openbridge)` extracts HAP globals and calls `openbridge.registerPlatform(...)`. OpenBridge's compat layer then instantiates `TuyaLocalPlatform` and emits `didFinishLaunching`.

### Accessory Pattern

Every device type follows the same pattern:

1. Extends `BaseAccessory` (which manages the `PlatformAccessory` lifecycle).
2. Implements `_registerPlatformAccessory()` to create HomeKit services and characteristics.
3. Reads/writes Tuya DataPoints (DPs) — numeric keys that map to device functions.
4. Receives DP updates from `TuyaAccessory` and pushes state changes to HomeKit.

### Device Type Registry

The `CLASS_DEF` map in `src/index.ts` maps lowercase type strings to classes:

```text
outlet, simplelight, rgbtwlight, rgbtwoutlet, twlight, multioutlet,
custommultioutlet, airconditioner, airpurifier, dehumidifier, convector,
garagedoor, simpledimmer, simpledimmer2, simpleblinds, simpleheater,
mappedheatpumpheater, circuitbreakermonitor, switch, fan, fanlight,
watervalve, oildiffuser
```

Users set the `type` field in their config (case-insensitive). The platform lowercases it and looks up the class.

### Config Validation

Zod schemas in `src/index.ts` validate config at startup:

- `TuyaDeviceConfigSchema` — validates each device entry (required `id`, `key`, `type`; `type` must be a known value).
- `NativePluginConfigSchema` — used in `setup()` for native mode. Accepts an optional `bridge` object with `name`, `username`, `pincode`, and `hapPort`.
- `OpenbridgePlatformConfigSchema` — used in the `TuyaLocalPlatform` constructor for compat mode.

### Communication Layer

- **TuyaAccessory** (`protocol/TuyaAccessory.ts`): Handles TCP connections to individual devices, encryption/decryption, and DP state management. Not a HomeKit accessory — it's the protocol client.
- **TuyaDiscovery** (`protocol/TuyaDiscovery.ts`): Listens for UDP broadcasts from Tuya devices on the LAN to discover IPs.

### Energy Monitoring

`accessories/EnergyCharacteristics.ts` is a factory that adds voltage, current, and power characteristics to outlets and outlet-like accessories that support energy readings.

## Conventions

- **Indentation**: 2 spaces everywhere (TS, JSON).
- **Line endings**: LF.
- **Naming**: PascalCase for classes, camelCase for variables/functions. Accessory files use the `{DeviceType}.accessory.ts` convention.
- **No external Tuya libraries**: The project implements the Tuya LAN protocol directly in `protocol/TuyaAccessory.ts` and `protocol/TuyaDiscovery.ts`.
- **OpenBridge types are inlined**: `@openbridge/sdk` is ESM-only. The `PluginLogger`, `PluginContext`, and `definePlugin` types/helpers are defined locally in `src/index.ts` to preserve the CommonJS build.
- **No `import()` of ESM packages at the top level**: All native OpenBridge types are inlined. Dynamic `import()` is only used for truly optional runtime loading.

## Adding a New Device Type

1. Create `src/accessories/{DeviceName}.accessory.ts` extending `BaseAccessory`.
2. Implement `_registerPlatformAccessory()` with the appropriate HomeKit services.
3. Export the class as `default` and add a re-export in `src/accessories/index.ts`.
4. Add the lowercase type key → class mapping in `CLASS_DEF` in `src/index.ts`.
5. Add the type to the `oneOf` array in `config.schema.json`.
6. Add any device-specific config parameters to the schema with appropriate `condition` functions.
7. Add the type key to the `VALID_TYPES` object in `src/index.ts` (used by Zod validation).

## Important Files to Keep In Sync

When making changes, ensure these files stay consistent:

| Change           | Files to update                                                                                                                   |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| New device type  | `src/accessories/{Name}.accessory.ts`, `src/accessories/index.ts`, `src/index.ts` (CLASS_DEF + VALID_TYPES), `config.schema.json` |
| New config param | `config.schema.json`, accessory class (read from `device.context`), Zod schemas in `src/index.ts`                                 |
| New dependency   | `package.json`, verify `npm run build` still passes                                                                               |

## Things to Avoid

- Do not add `"type": "module"` to `package.json` — the project is CommonJS. OpenBridge's compat loader uses `require()`.
- Do not import from `@openbridge/sdk` or `@openbridge/core` at the top level — they are ESM-only packages and will break the CommonJS build. Define OpenBridge types locally instead.
- Do not import from `openbridge` in accessory files at the top level — HAP types are injected at runtime via the registration function.
- Do not expose circuit breaker switch control to HomeKit — `CircuitBreakerMonitorAccessory` is intentionally read-only for safety.
- Do not create singleton `TuyaAccessory` instances — each device gets its own connection.
- Do not remove the dual-export at the bottom of `src/index.ts` — both loading paths depend on it.
